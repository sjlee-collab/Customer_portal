## 1. 고객 지원 포털 구축 프로젝트 제안

- **대표님**이 약 2년간 논의되었던 고객 지원 사이트 구축의 필요성을 제기하며 회의를 시작함. 과거 외주를 고려했으나 비용 및 관리 문제로 무산되었으며, 내부 개발 툴인 '바이브코딩'을 활용한 자체 구축을 제안함.
- **핵심 기능 요구사항**:
  - 고객의 라이선스 및 계약 정보 확인
  - 자료 다운로드 (배포 가이드 문서, 사용자 교육 자료 등)
  - 기술 지원 요청 및 이력 조회
  - FAQ 조회
- **회의 목표**: '바이브코딩'으로 구현 가능 여부, 예상 소요 기간, 담당자 지정을 논의하고자 함.

## 2. 데이터 연동 및 관리 방식 논의

- **원이쌤**은 구축 자체는 긍정적이나, 포털이 활발히 사용되지 않을 가능성과 데이터 연동의 선행 필요성을 지적함. 특히 라이선스 및 계약 정보가 **Salesforce**에 있으므로 연동이 필수적이라고 강조함.
- 데이터 소스에 대한 이견이 있었음. **대표님**은 **Salesforce** 데이터가 불완전할 것이라며 원천 데이터의 문제를 지적했으나, **이수민**, **강원이**, **이준호**는 수동 입력 방식의 비효율성과 휴먼 에러 가능성을 들며 **Salesforce**와의 API 연동 및 동기화가 필수적이라고 강력히 주장함.
- **이준호**는 **CRM(Salesforce)** 데이터를 기반으로 하고, 필요한 추가 정보만 덧붙이는 형태가 되어야 한다고 정리함.
- **대표님**은 기술적 회의론을 표하면서도, **Salesforce** 연동이 가능하다면 그렇게 진행하라고 한발 물러섬.
- **이준호**는 **Salesforce API** 연동이 이미 다른 시스템에서 사용 중이라 기술적으로 어려운 작업이 아니라고 확인함.

## 3. 사용자 계정 관리 및 보안 강화 방안

- **사용자 정책**: **대표님**은 이 포털이 계약 관계가 있는 고객 전용임을 명확히 함. 공개적인 회원가입 기능은 없으며, 내부에서 계정을 생성하여 고객에게 전달하거나 고객사 내 마스터 계정 관리자가 사용자를 추가하는 방식을 제안함.
- **보안 문제 제기**: **수민쌤**이 계약 정보와 같은 민감 정보를 웹페이지에 직접 표시할 경우 해킹 시 심각한 문제가 발생할 수 있다고 지적함.
- **해결 방안**: 민감 정보는 포털에 직접 노출하는 대신, 사용자가 요청하면 담당자 이메일로 발송하는 방식으로 처리하기로 즉시 합의함.

## 4. 기술 지원 프로세스 및 이력 관리

- **요청 처리 방식**: **Speaker 1**은 고객이 포털을 통해 기술 지원을 요청하면 해당 내역이 **Slack**이나 이메일로 담당자에게 전달되고, 모든 요청과 처리 이력이 시스템에 기록되어야 한다고 강조함.
- **기존 시스템 연동**: **Speaker 2**는 현재 **Slack**으로 접수된 요청이 **Notion**에 자동으로 등록되는 워크플로우를 사용 중이며, **Notion API**를 통해 이력을 포털에 연동할 수 있다고 설명함. 단, 고객 공개를 위해서는 내부용으로 관리되던 내용을 정리할 필요가 있다고 덧붙임.
- **이력 관리의 중요성**: **Speaker 1**은 고객이 자신의 지원 요청 이력을 직접 확인할 수 있어야 '지원을 제대로 받지 못했다'는 불만을 줄일 수 있다고 역설함. **카카오**의 사례를 들며, 실제 지원 횟수와 고객이 체감하는 지원 수준 간의 간극을 시스템으로 해소할 수 있다고 강조함.

## 5. 프로젝트 범위 및 단계별 접근 전략

- **단계별 개발 제안**: **Speaker 1**은 프로젝트 범위가 비대해져 무산되는 것을 막기 위해 단계별 접근을 제안함. **1단계**에서는 고객 정보 관리와 기술 지원 요청 접수 등 핵심 기능만 갖춘 기본 '창고' 역할의 프로토타입을 신속하게 개발하자고 주장함.
- **프로젝트 목표**: 현재 **300개**에 달하는 고객사를 전화나 개별 연락으로 관리하는 방식에서 벗어나, 체계적인 관리 시스템의 뼈대를 우선적으로 구축하는 것을 목표로 함.
- **비계약 고객 처리**: **Speaker 4**는 **PoC** 진행 고객 등 모든 고객에게 포털 접근 권한을 부여하여 일관된 경험을 제공하고, 계약 유형에 따라 기능 접근 범위를 차등화하자고 제안함. 이에 **Speaker 1**은 해당 시스템은 기본적으로 라이선스나 서비스 계약이 있는 고객을 위한 것이라고 선을 그음.

## 6. TF팀 구성 및 실행 계획

- **개발 방식**: **Speaker 1**은 '바이브코딩'을 활용해 1차 버전을 빠르게 만들 것을 재차 강조하며, 잦은 중간 보고를 통해 프로젝트 지연을 방지하자고 함.
- **TF팀 구성**:
  - **개발**: **구상임(Speaker 4)**이 개발 인력(1~2명)을 지정하기로 함. **Speaker 2**는 DB 설계를 고려해 2명 협업을 제안함.
  - **요구사항 정의**: **마수민**, **강원이**, **윤수민**이 담당하며, **마승희**를 반드시 참여시키라고 **Speaker 1**이 지시함.
  - **보안 검토**: **구상임(Speaker 4)**이 '경호쌤'을 TF에 포함시켜 보안 검토를 진행하기로 함.
- **기술 스택**: 기존 HR 시스템 등에서 사용 중인 **AWS RDS**의 **PostgreSQL DB**를 활용할 수 있다고 **구상임(Speaker 4)**이 언급함.
- **최종 지시**: **Speaker 1**은 프로젝트 시작을 공식화하며, 프로토타입 제작 후 중간 보고를 할 것을 지시하며 회의를 마무리함.

## Key Decisions

- **1. 고객 지원 포털 자체 구축**: 지난 2년간 논의되었으나 외주 비용 및 관리 부담으로 지연된 고객 지원 포털을 내부 개발 툴 '바이브코딩'을 활용해 자체적으로 구축하기로 결정함.
- **2. 민감 정보 처리 방안**: 보안 강화를 위해 계약 정보 등 민감 정보는 포털에 직접 표시하지 않고, 사용자가 요청 시 담당자 이메일로 발송하는 방식으로 처리하기로 함.
- **3. 단계별 개발 접근**: 프로젝트 무산을 방지하고 빠른 결과물 도출을 위해, 1단계에서는 핵심 기능을 갖춘 프로토타입을 신속하게 개발하고 이후 점진적으로 기능을 확장하기로 함.
- **4. 사용자 접근 정책**: 포털 사용자는 계약 고객으로 한정하며, 일반 사용자의 회원가입 기능 없이 내부에서 계정을 생성하여 전달하는 폐쇄적인 방식으로 운영하기로 함.
- **5. TF팀 구성 및 역할 분담**: 프로젝트의 공식적인 시작을 위해 TF팀을 구성함. **구상임**이 개발 인력을 지정하고, **마수민**, **강원이**, **윤수민**, **마승희**가 요구사항 정의를 담당하기로 함.
